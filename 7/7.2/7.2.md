# 7.2. Patrones de Componente y Conector

## Introducción

En **Turi**, los patrones de componente y conector seleccionados aseguran una arquitectura modular y escalable, que facilita la comunicación eficiente entre los diferentes componentes del sistema. Los patrones seleccionados son **Broker (API Gateway)**, **MVC (backend)** y **Cliente-Servidor**, cada uno desempeñando un papel clave en la estructura de la plataforma.

---

### Broker (API Gateway)

El patrón **Broker** es implementado a través de un API Gateway que actúa como intermediario entre los clientes (usuarios de la plataforma móvil) y los servicios backend. Este patrón simplifica la comunicación, mejora la seguridad y facilita la integración con múltiples módulos y servicios.

#### Funciones clave:
- **Enrutamiento de solicitudes**: Redirige las solicitudes de los clientes al microservicio correspondiente.
- **Gestión de seguridad**: Implementa autenticación, autorización y protección contra ataques como DDoS.
- **Transformación de datos**: Adapta el formato de las solicitudes/respuestas para cumplir con las necesidades de los clientes y los servicios backend.
- **Consolidación de servicios**: Combina respuestas de múltiples servicios backend en una sola respuesta para el cliente.

#### Ventajas:
- Reducción de complejidad en la interacción cliente-servidor.
- Centralización de políticas de seguridad y monitoreo.
- Mayor flexibilidad para escalar y modificar servicios de backend sin afectar al cliente.

---

### MVC (Backend)

El backend de **Turi** sigue el patrón **Modelo-Vista-Controlador (MVC)**, que organiza la lógica del sistema en componentes claramente separados para mejorar la mantenibilidad y facilitar la colaboración entre equipos.

#### Componentes:
- **Modelo**: Maneja los datos y la lógica de negocio, como la creación de itinerarios y la gestión de usuarios.
- **Vista**: Proporciona las respuestas al cliente, normalmente en formato JSON o HTML, dependiendo del consumidor.
- **Controlador**: Gestiona las solicitudes del cliente, interactúa con el modelo y decide qué datos enviar a la vista.

#### Ventajas:
- Separación clara de responsabilidades, reduciendo el acoplamiento entre componentes.
- Mayor facilidad para realizar pruebas y mantenimiento del código.
- Escalabilidad al poder trabajar en cada componente de manera independiente.

---

### Cliente-Servidor

La arquitectura **Cliente-Servidor** es la base de la interacción entre la aplicación móvil y el backend de **Turi**. Este patrón establece una relación en la que el cliente (la app móvil) solicita servicios y el servidor (backend) responde con los datos o acciones necesarias.

#### Características principales:
- **Cliente**: Solicita datos o servicios (por ejemplo, la búsqueda de lugares turísticos o la creación de itinerarios).
- **Servidor**: Procesa las solicitudes, interactúa con la base de datos y envía las respuestas correspondientes.
- **Comunicación**: Se realiza a través de protocolos estándar como HTTPS para garantizar la seguridad.

#### Ventajas:
- Claridad en la interacción entre el frontend y el backend.
- Independencia de la implementación del cliente y del servidor, permitiendo iteraciones rápidas en ambos lados.
- Escalabilidad horizontal al poder desplegar múltiples instancias del servidor.

---

## Relación entre los Patrones

La combinación de **Broker (API Gateway)**, **MVC (Backend)** y **Cliente-Servidor** proporciona a **Turi** una arquitectura sólida y flexible. El API Gateway simplifica la comunicación y agrega una capa de seguridad, mientras que el patrón MVC organiza la lógica backend, y el modelo Cliente-Servidor asegura una interacción eficiente entre la aplicación y el servidor. Juntos, estos patrones optimizan el rendimiento y la capacidad de mantenimiento de la plataforma.
